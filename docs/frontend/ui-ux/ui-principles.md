# UI Principles

화면 전반에 걸쳐 일관된 UI를 유지하기 위한 구성 원칙.

---

## 스켈레톤 UI 표시 기준

스켈레톤 UI는 **화면 레이아웃이 미리 예측 가능한 경우**에만 사용한다.

| 상황 | 처리 방식 |
|------|-----------|
| 페이지 초기 진입 시 데이터 로딩 | 스켈레톤 UI |
| 테이블/목록 데이터 로딩 | row 형태 스켈레톤 |
| 카드 레이아웃 데이터 로딩 | 카드 형태 스켈레톤 |
| 버튼 클릭 후 짧은 액션 | 버튼 내 로딩 스피너 |
| 전체 페이지를 대체할 수 없는 경우 | 스피너 또는 로딩 텍스트 |

스켈레톤은 실제 컴포넌트의 레이아웃 구조를 반영해야 한다. 단순 회색 박스 나열은 사용하지 않는다.

---

## 빈 화면 처리

데이터가 없는 경우 반드시 빈 상태(Empty State) UI를 제공한다. 빈 화면은 세 가지 요소로 구성한다.

1. **아이콘 또는 일러스트** — 콘텍스트를 시각적으로 전달
2. **안내 문구** — 왜 비어있는지 간결하게 설명
3. **액션 버튼** (선택) — 데이터를 추가할 수 있는 CTA 제공

```
[아이콘]
등록된 카테고리가 없습니다.
[카테고리 추가]
```

### 빈 상태 문구 원칙

- "데이터가 없습니다" 같은 기술적 표현 대신 사용자 관점의 문구 사용
- 검색/필터 결과가 없는 경우와 데이터 자체가 없는 경우를 구분하여 다른 문구 사용

| 상황 | 문구 예시 |
|------|-----------|
| 데이터 자체 없음 | "등록된 {이름}이 없습니다." |
| 검색 결과 없음 | "검색 결과가 없습니다. 다른 키워드로 검색해보세요." |
| 필터 결과 없음 | "해당 조건에 맞는 항목이 없습니다." |

## 버튼 위치 일관성

### 페이지/섹션 기본 액션

- **주요 CTA (생성, 저장)**: 우측 상단 또는 우측 하단
- **취소**: CTA 버튼 좌측에 배치
- **위험 액션 (삭제)**: 좌측 하단 또는 주요 액션과 명확히 분리

### 폼 하단 버튼 배치

```
[삭제]                    [취소] [저장]
```

- 구분이 필요한 destructive 버튼은 반대편(좌측)에 배치
- 취소와 확인 버튼은 항상 쌍으로 붙여 배치
- 버튼 순서: 취소 → 확인 (좌 → 우)

### 모달 하단 버튼 배치

```
                         [취소] [확인]
```

- 모달 내 버튼은 항상 우측 정렬
- confirm dialog의 destructive 버튼은 우측 마지막에 배치하되 danger 색상 적용

## 섹션 간 여백 기준

Tailwind CSS spacing scale 기준.

| 구분 | 클래스 | 사용처 |
|------|--------|--------|
| 컴포넌트 내부 요소 간 | `gap-2` / `space-y-2` (8px) | 라벨-입력, 아이콘-텍스트 |
| 폼 필드 간 | `space-y-4` (16px) | 입력 필드 사이 |
| 섹션 내 그룹 간 | `space-y-6` (24px) | 관련 필드 그룹 사이 |
| 페이지 내 섹션 간 | `space-y-8` (32px) | 독립 섹션 사이 |
| 페이지 상단 여백 | `pt-6` (24px) | 페이지 최상단 |

페이지 전체 좌우 패딩은 `px-6`(모바일) / `px-8`(데스크톱)으로 통일한다.

## 모바일/데스크톱 대응 기준

Backoffice는 **데스크톱 우선** 설계이나, 태블릿 수준의 반응형은 지원한다.

| 구분 | 기준 |
|------|------|
| 데스크톱 | 1024px 이상 — 사이드바 + 콘텐츠 레이아웃 |
| 태블릿 | 768px ~ 1023px — 사이드바 축소 또는 오버레이 |
| 모바일 | 767px 이하 — 필수 기능만 지원, 레이아웃 단순화 |

### 반응형 처리 원칙

- 테이블은 모바일에서 가로 스크롤 허용 (컬럼 숨기기 대신)
- 모달은 모바일에서 바텀 시트 방식으로 전환 가능
- 중요하지 않은 컬럼은 `hidden md:table-cell`로 처리
- 버튼 텍스트는 모바일에서 아이콘만 표시하는 방식 허용 (`hidden sm:inline`)

## CTA 버튼 우선순위

한 화면에서 버튼의 시각적 강조 수준은 계층을 가져야 한다.

| 단계 | 스타일 | 수량 제한 | 용도 |
|------|--------|-----------|------|
| Primary | 채운 배경색 (brand color) | 화면당 1개 | 가장 중요한 단일 행동 |
| Secondary | 외곽선(outline) 또는 연한 배경 | 2~3개 | 보조 행동 |
| Tertiary | 텍스트 버튼 (배경 없음) | 제한 없음 | 부가 액션, 취소 |
| Danger | 채운 빨간색 배경 | 화면당 1개 | 삭제 등 destructive |

- Primary 버튼이 두 개 이상이면 사용자는 어떤 행동이 중요한지 판단하기 어렵다.
- 페이지의 목적을 대표하는 단 하나의 Primary 액션을 정의한다.

## 색상/텍스트 강조 기준

### 색상 사용 원칙

| 목적 | 색상 |
|------|------|
| 브랜드/Primary 액션 | brand color (blue 계열) |
| 성공/완료 상태 | green 계열 |
| 경고/주의 | yellow/amber 계열 |
| 오류/위험 | red 계열 |
| 비활성/보조 텍스트 | gray 계열 |

- 색상만으로 상태를 전달하지 않는다. 아이콘 또는 텍스트를 함께 사용한다.
- 배경이 어두운 경우 텍스트 대비 비율은 WCAG AA 기준(4.5:1) 이상을 유지한다.

### 텍스트 강조 기준

| 위계 | 클래스 예시 | 사용처 |
|------|-------------|--------|
| 페이지 제목 | `text-2xl font-bold` | 페이지 상단 H1 |
| 섹션 제목 | `text-lg font-semibold` | 카드/섹션 제목 |
| 본문 | `text-sm text-gray-900` | 일반 콘텐츠 |
| 보조 텍스트 | `text-sm text-gray-500` | 설명, 힌트 |
| 에러 텍스트 | `text-sm text-red-600` | 유효성 오류 메시지 |

- 강조를 위해 굵기(`font-semibold`, `font-bold`)와 색상을 동시에 사용하지 않는다. 하나만 선택한다.
- 밑줄은 링크 텍스트에만 사용한다.
