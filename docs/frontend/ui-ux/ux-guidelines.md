# UX Guidelines

사용자 인터랙션 전반에 걸쳐 일관된 경험을 제공하기 위한 원칙.

---

## 삭제는 반드시 Confirm Dialog 제공

삭제 액션은 항상 사용자에게 confirm dialog를 통해 의도를 재확인한다.

- 삭제 버튼 클릭 → 즉시 삭제 금지
- confirm dialog에는 삭제 대상이 무엇인지 명시
- 되돌릴 수 없는 삭제임을 dialog 내 문구로 안내

```
"'여름 신상품' 카테고리를 삭제하시겠습니까?
삭제된 데이터는 복구할 수 없습니다."

[취소]  [삭제]
```

단, 아래의 경우는 confirm dialog 없이 즉시 처리 가능하다.

- Undo가 가능한 임시 삭제 (삭제 후 일정 시간 내 되돌리기 가능)
- 초안/임시저장 상태 데이터 (게시되지 않은 상태)

---

## 저장 중 중복 클릭 방지

저장/제출 버튼은 클릭 후 응답이 완료될 때까지 비활성화하거나 로딩 상태로 전환한다.

- 버튼에 `disabled` 속성 추가
- 버튼 텍스트 또는 아이콘을 로딩 상태로 변경
- 폼 전체를 `pointer-events-none`으로 처리하는 방식은 지양

```
저장 중: [저장] → [● 저장 중...]
완료 후: [저장]으로 복귀
```

네트워크 지연이 있어도 사용자가 "버튼이 눌렸는지" 인지할 수 있도록 즉각적인 피드백(버튼 상태 변경)을 제공한다.

---

## 성공/실패 피드백 제공

모든 사용자 액션은 성공 또는 실패 결과를 명확히 알려야 한다.

| 결과 | 피드백 방식 |
|------|-------------|
| 저장 성공 | 토스트 메시지 (green) |
| 삭제 성공 | 토스트 메시지 (green) + 목록 반영 |
| 일반 오류 | 토스트 메시지 (red) |
| 유효성 오류 | 입력 필드 근처 인라인 에러 메시지 |
| 권한 오류 | 오류 내용을 안내하는 토스트 또는 페이지 |

- 성공 토스트는 자동으로 닫힌다 (3초 기준).
- 실패 토스트는 사용자가 직접 닫거나 더 긴 시간 동안 유지한다 (5초 기준).
- 오류 메시지는 기술적 코드가 아닌 사용자가 이해할 수 있는 문구로 작성한다.

---

## 로딩 상태 명확히 표시

사용자가 요청이 처리 중임을 알 수 있도록 반드시 로딩 상태를 표시한다.

| 상황 | 처리 방식 |
|------|-----------|
| 페이지 진입 시 데이터 로딩 | 스켈레톤 UI |
| 버튼 클릭 후 응답 대기 | 버튼 로딩 상태 |
| 목록 필터/검색 변경 | 테이블 row 스켈레톤 또는 오버레이 |
| 파일 업로드 | 프로그레스 바 또는 퍼센트 표시 |

"로딩 중" 상태가 없다면 사용자는 요청이 실패했다고 오해하거나 버튼을 반복 클릭한다.

---

## Destructive Action 명확히 구분

위험한 액션(삭제, 비활성화, 초기화 등)은 일반 액션과 시각적으로 명확히 구분한다.

- 삭제 버튼은 red 계열 색상 적용 (`text-red-600`, `bg-red-600`)
- 버튼 레이블에 동작을 명시: "삭제" / "비활성화" / "초기화" (추상적인 "확인" 사용 금지)
- 일반 저장 버튼과 같은 위치에 배치하지 않는다 (분리 배치)

```
[삭제]                    [취소] [저장]
 ↑ red, 좌측              ↑ 우측 정렬
```

---

## 되돌릴 수 없는 작업은 사전 안내

완료 후 되돌릴 수 없는 작업은 실행 전에 사용자에게 명확히 고지한다.

안내가 필요한 작업 예시:
- 영구 삭제
- 발행된 콘텐츠 비공개 전환 (외부 영향 있음)
- 대량 일괄 처리

안내 방법:
1. confirm dialog 내 "이 작업은 되돌릴 수 없습니다." 문구 포함
2. 작업 버튼 hover 시 tooltip으로 영향 범위 안내 (선택)
3. 영향을 받는 데이터 범위를 dialog에 명시 (예: "3개의 항목이 삭제됩니다.")

---

## 폼 유효성 에러는 입력 필드 근처에 표시

유효성 오류 메시지는 문제가 발생한 필드 바로 아래 인라인으로 표시한다.

```
[상품명 입력]
※ 상품명은 필수 항목입니다.   ← 필드 바로 아래, red 텍스트
```

- 페이지 상단에 에러 목록을 모아두는 방식은 사용하지 않는다.
- 제출 실패 후 에러가 있는 첫 번째 필드로 스크롤 이동 또는 포커스 이동.
- 에러 발생 조건: 제출 시도 후 또는 필드 blur 시점 (입력 중 실시간 표시는 지양).
- 에러가 해결되면 메시지를 즉시 제거한다.

---

## 네트워크 실패 시 재시도 UX 제공

네트워크 오류 또는 서버 오류 발생 시 사용자가 직접 재시도할 수 있는 수단을 제공한다.

| 상황 | 처리 방식 |
|------|-----------|
| 페이지 데이터 로딩 실패 | 에러 메시지 + [다시 시도] 버튼 |
| 목록 조회 실패 | 에러 상태 UI + [새로고침] 버튼 |
| 저장/수정 실패 | 실패 토스트 + 폼 입력값 유지 (초기화 금지) |
| 파일 업로드 실패 | 실패 안내 + [다시 업로드] 버튼 |

- 실패 시 입력한 데이터를 초기화하지 않는다. 사용자가 다시 입력해야 하는 상황을 만들지 않는다.
- 재시도 버튼 문구는 "다시 시도" 또는 맥락에 맞는 동사형으로 작성한다.
- 자동 재시도는 사용자에게 알리지 않고 처리할 수 있으나, 최종 실패는 반드시 명시한다.
