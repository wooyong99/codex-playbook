# Design Documents (TDD)

기능·서브시스템의 **기술 설계 문서**(Technical Design Document). 아키텍처 판단 근거, 계층 분리, 트랜잭션 경계, 예외·실패 처리, 동시성·정합성, 확장 가능성 등 **코드만으로는 읽히지 않는 설계 의도**를 담는다.

---

## 언제 작성하는가

- 여러 도메인·모듈에 걸친 **새 기능**을 설계할 때
- 기존 기능의 **중대한 구조 변경**이 필요할 때
- 동시성·정합성·트랜잭션 경계에 **비자명한 선택**이 필요한 때
- 향후 유지보수·확장 시 **결정의 맥락**을 남겨야 할 때

단일 CRUD·단순 UseCase 추가는 **작성하지 않는다.** 과도한 문서화는 오히려 비용이다.

---

## 파일명 규칙

```
tdd-<feature-slug>.md
```

예: `tdd-product-wishlist.md`, `tdd-payment-settlement.md`

---

## 문서 헤더

파일 상단에 아래 메타 정보를 인용 블록으로 둔다.

```
> 작성일: YYYY-MM-DD
> 상태: Draft | Reviewing | Approved | Superseded
> 대상 모듈: :app:{channel}, :core:application, :core:domain, :infra:storage 등
```

## 표준 섹션

아래 9개 섹션 + 부록을 기본 골격으로 한다. 기능 성격에 맞게 가감하되, 빈 섹션은 두지 않는다.

1. **설계 배경 및 목적**
   - 1.1 배경
   - 1.2 설계 목표 (번호 매긴 리스트)
   - 1.3 설계 비목표 — 의도적으로 다루지 않는 범위를 명시
   - 1.4 기술적 제약사항

2. **현행 시스템 분석**
   - 2.1 관련 도메인 구조 (ASCII 다이어그램 권장)
   - 2.2 현재 처리 흐름
   - 2.3 현행 스키마 분석

3. **아키텍처 설계**
   - 3.1 계층별 책임 분배 (표: 계층·구성 요소·책임·설계 근거)
   - 3.2 처리 흐름 (번호 매긴 단계별 의사코드)
   - 3.3 설계 대안 분석 (표: 대안·장점·단점·채택 여부·사유)

4. **도메인 모델 설계**
   - 4.1 애그리거트 경계
   - 4.2 도메인 모델 상세 (애그리거트별 역할·불변식·주요 행위·상태 전이)
   - 4.3 데이터 스키마 설계 (DDL 스니펫)
   - 4.4 데이터 변환 흐름 (HTTP → DTO → Command → Domain → Entity → DB)

5. **트랜잭션 설계**
   - 5.1 트랜잭션 경계 (표: 연산·시작점·범위·격리 수준·사유)
   - 5.2 정합성 보장 전략
   - 5.3 이벤트 처리

6. **예외 및 실패 처리**
   - 6.1 예외 분류 (표: 예외 유형·ErrorCode·발생 조건·HTTP 상태·사용자 메시지)
   - 6.2 실패 시나리오 및 복구 전략 (표: 시나리오·발생 가능성·영향·복구)
   - 6.3 멱등성 보장

7. **동시성 및 성능**
   - 7.1 동시성 제어 (경합 지점별 제어 방식)
   - 7.2 성능 고려사항
   - 7.3 확장 가능성 (열어둔 포인트 / 의도적으로 닫아둔 제약)

8. **변경 파일 목록**
   - 파일 경로, 모듈, 변경 유형(신규/수정), 설명을 표로 나열

9. **검증 계획**
   - 시나리오별 테스트 유형(단위/통합)·검증 내용·예상 결과

**부록**: 요청/응답 예시, 추가 다이어그램, 외부 인용 등

## 검토 에이전트의 취급

`architecture-reviewer` 에이전트는 이 디렉토리를 **Source of Truth에서 제외**한다.
설계 의도를 참고하되 *준수 규칙*으로 강제하지 않는다. 강제할 규칙은 `architecture/*` 또는 `policies/*`에 규정으로 올려야 한다.
