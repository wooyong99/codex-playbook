# Architecture 문서 작성 컨벤션

## 원칙

- 문서 구조는 예측 가능해야 한다. 모든 세부 컨벤션 문서가 동일한 섹션 순서를 따르면 어느 문서를 열어도 원하는 정보를 같은 위치에서 찾을 수 있다.
- 내용은 한 곳에만 있다. 같은 규칙이 여러 문서에 중복 기술되면 변경 시 모두 업데이트해야 하고, 불일치가 발생한다.
- 상위 문서는 링크만 제공한다. 세부 구현은 하위 문서에 위임하고, 상위 문서는 전체 구조 파악을 돕는 인덱스 역할에 집중한다.
- 두괄식으로 작성한다. 결론(규칙)을 먼저 제시하고 근거·예시를 나중에 전개하면 필요한 정보를 빠르게 찾을 수 있다.

---

`docs/backend/architecture/` 하위 레이어별 가이드라인·세부 컨벤션 문서 작성 시 준수해야 할 구조·템플릿·링크 규칙을 정의한다.

---

## 핵심 규칙

**각 레이어 디렉토리는 "최상위 가이드라인 문서 1개(보편 원칙) + 구성 요소별 세부 컨벤션 문서 N개"로 구성한다. 최상위 문서는 보편 원칙·금지 규칙·전략 링크를 담고, 세부 문서는 두괄식 템플릿을 따른다.**

전략 종속 컨벤션은 `strategies/`에 분리한다. 중복된 내용은 한쪽에만 두고 다른 쪽은 "참고 링크"로 연결한다. 앵커 링크는 사용하지 않고 파일 링크 + "섹션 이름 참고" 패턴을 쓴다.

---

## 디렉토리 구조

```
docs/backend/architecture/
├── documentation-convention.md              ← 이 문서 (문서 작성 규칙)
├── {layer}/
│   ├── {layer}-layer-guidelines.md          ← 최상위: 보편 원칙 + 전략 링크
│   ├── {component}-convention.md            ← 세부: 전략 독립 컨벤션 (있을 경우)
│   └── strategies/                          ← 구현 전략 종속 컨벤션 (있을 경우)
│       ├── README.md                        ← 전략 정의 + 커스터마이징 템플릿
│       └── {component}-convention.md        ← 전략별 세부 컨벤션
```

`strategies/` 하위 디렉토리는 모듈의 내부 구현 방식이 프로젝트마다 달라질 수 있을 때 사용한다. 전략이 고정된 모듈은 컨벤션 파일을 최상위 문서와 같은 디렉토리에 flat하게 둔다.

| 레이어 | 최상위 |
|--------|-------|
| application | application-layer-guidelines.md |
| domain | domain-layer-guidelines.md |
| storage | storage-layer-guidelines.md |
| external | external-layer-guidelines.md |
| app | app-layer-guidelines.md |

## 최상위 문서 (`{layer}-layer-guidelines.md`) 템플릿

최상위 문서는 **보편 원칙**을 담는다. 전략 종속 규칙·예시·판단 기준은 `strategies/`에 위임하고, 반복하지 않는다.

### 필수 섹션 순서

```markdown
# {Module} Layer Guidelines

## {Module} 계층의 본질적 책임

{계층이 존재하는 이유와 핵심 책임을 2~4줄로 서술}

1. **{책임 1}**: {설명}
2. **{책임 2}**: {설명}

---

## 반드시 지켜야 할 규칙

- **R1. {규칙 제목}** — {규칙 설명}
- **R2. {규칙 제목}** — {규칙 설명}

---

## 금지 규칙 / 안티패턴

- **{안티패턴 제목}** — {왜 문제인지 한 줄 설명}
- **{안티패턴 제목}** — {왜 문제인지 한 줄 설명}

---

## 이 프로젝트의 로컬 컨벤션

{내부 구현 전략이 있는 경우} 내부 구성 방식은 프로젝트 환경·팀 선호에 따라 자유롭게 선택할 수 있다. 어떤 전략을 선택하든 R1–RN은 반드시 지킨다. 역할 정의, 전략 선택 기준, 이 프로젝트의 선택 → [`strategies/`](strategies/README.md)

### 공통 규칙

**{규칙 항목}**: {설명 + 관련 정책 링크}

### Post-Work Verification

구현 완료 후 생성·수정한 파일을 직접 읽어 아래 각 문서의 체크리스트를 대조한다.

이 프로젝트의 체크리스트 문서 목록 → [`strategies/`](strategies/README.md)
```

> 내부 구현 전략이 없는 단순 모듈은 첫 줄(전략 선택 안내)을 생략하고, 세부 컨벤션 링크를 공통 규칙 아래에 직접 나열한다.

### 금지 사항

- **세부 컨벤션 문서의 규칙을 최상위에서 재기술하지 않는다.** 보편 원칙(R1–RN)만 여기에 두고, 컴포넌트별 세부 규칙은 하위 문서에 위임한다.
- **전략 종속 컨벤션을 최상위 문서에 직접 포함하지 않는다.** `strategies/`로 분리하고 링크만 제공한다.
- **체크박스를 최상위 문서 본문에 두지 않는다.** Post-Work Verification은 하위 문서 링크로만 구성한다.
- **구성 요소별 예시 코드를 최상위에 두지 않는다.** 예시는 해당 세부 컨벤션 문서에만.

## 세부 컨벤션 문서 (`{component}-convention.md`) 템플릿

세부 문서는 **두괄식(결론 먼저) + 일관된 섹션 순서**를 지킨다.

### 표준 섹션 순서

```markdown
# {Component} 컨벤션

---

## 핵심 규칙

**{한 문장 결론 — 굵게}**

{왜 이 규칙이 필요한지 2~3줄 설명 + 관련 문서 링크}

---

## 네이밍 규칙 (해당되는 경우)

| 항목 | 패턴 | 예시 |

---

## {주제별 상세 섹션 1}

**규칙: {섹션별 결론 — 굵게}**

{설명}

```kotlin
// ❌ / ✅ 예시
```

---

## {주제별 상세 섹션 2}

...

---

## 판단 기준 (해당되는 경우, 각 주제 내부에 녹이거나 별도 섹션)

| 상황 | 방식 |

---

## 금지 사항

- {안티패턴 1}
- {안티패턴 2}

---

## 체크리스트

- [ ] {규칙 1}을 준수했는가?
- [ ] {규칙 2}를 준수했는가?
```

### 섹션별 규칙

| 섹션 | 필수 | 규칙 |
|------|-----|------|
| 핵심 규칙 | **필수** | 첫 섹션. **굵은 한 문장 결론**으로 시작. 이어서 2~3줄 이유 설명 |
| 네이밍 규칙 | 해당 시 | 별도 섹션으로 분리 (표 형식). 본문 중간에 녹이지 않음 |
| 주제별 상세 | 필수 | 각 섹션도 두괄식 (`**규칙: ...**`) |
| 예시 | 필수 | ❌/✅ 대비를 코드 블록으로. Kotlin 하이라이트 |
| 판단 기준 | 권장 | 표 형식. 주제별 섹션에 녹여도 되고 별도 섹션도 가능 |
| 금지 사항 | **필수** | 안티패턴을 불릿 목록으로 명시 |
| 체크리스트 | **필수** | 문서 끝. 체크박스 형식. 구현 후 자기 검증용 |

### 금지 사항

- **번호 섹션(`## 1.`, `## 2.` …)을 사용하지 않는다.** 번호 없는 섹션 제목만 사용한다.
- **"핵심 원칙 요약" 표로 시작하지 않는다.** 첫 섹션은 반드시 **"핵심 규칙" 굵은 한 문장 결론**으로 시작한다.
- **상위 모듈 문서와 중복되는 섹션(파일 구조 등)을 두지 않는다.** 상위에 집약하고 하위에서는 링크만.
- **다른 세부 컨벤션 문서가 소유한 내용을 재기술하지 않는다.** 예: `domain-model-convention`은 예외 처리 상세를 `exception-convention.md`로 링크만 건다.

## 공통 작성 규칙

### 두괄식 (결론 먼저)

모든 섹션은 **굵은 한 문장 결론**으로 시작하고, 이어서 이유·예시·세부를 전개한다.

```markdown
## 쿼리 작성 규칙

**규칙: 쿼리 작성 순서를 고정하고, 조합 가능성을 기준으로 설계한다.**

`select → from → join → where → ...` 순서를 고정하면 의도가 선언적으로 드러난다.

(예시 코드 · 판단 기준 · 세부 규칙 이어짐)
```

### 예시 코드

- Kotlin 코드 블록으로 작성 (` ```kotlin ... ``` `)
- 잘못된 예시(`❌`)와 올바른 예시(`✅`)를 **쌍으로** 제시하여 비교 가능하게 한다
- 예시는 **최소 가독 단위**로 자른다 — 한 섹션당 20~40줄 이내가 적당

### 표(Table) 활용

판단 기준·네이밍 규칙·의존 가능/불가 등은 **표 형식**으로 정리한다. 산문보다 스캔 가능성이 높아진다.

```markdown
| 조건 | 방식 | 예시 |
|------|------|------|
| 단일 UseCase | UseCase 내 private 메서드 | ... |
| 2개 이상 UseCase 재사용 | Flow로 추출 | ... |
```

### 금지 사항과 체크리스트의 관계

- **금지 사항**: 서술형(완전한 문장) — "~을 하지 않는다"
- **체크리스트**: 의문형(질문) — "~을 준수했는가?"
- 같은 규칙을 양쪽에 적되, 형식이 대칭되어야 한다.

## 링크 규칙

### 기본 원칙

**앵커 링크(`#header-name`)를 사용하지 않는다. 파일 링크만 사용하고, 필요 시 "섹션 이름 참고"를 텍스트로 덧붙인다.**

IntelliJ·VS Code·GitHub 각각의 Markdown 렌더러가 한글 헤더 앵커와 HTML `<a id="...">` 앵커를 일관되게 처리하지 않아, **앵커 링크는 "이동할 헤더를 찾을 수 없습니다"를 자주 일으킨다.** 파일 링크만 쓰면 모든 환경에서 안정적으로 동작한다.

### 패턴

```markdown
✅ > 검증 위치 판단 기준은 [validator-convention.md](validator-convention.md) "Validator 호출 위치" 섹션 참고
✅ > Flow 내부 관련 금지 사항은 [flow-convention.md](flow-convention.md) "금지 사항" 섹션 참고

❌ > [validator-convention.md#validator-호출-위치](validator-convention.md#validator-호출-위치)
❌ <a id="checklist"></a>
❌ [validator-convention.md#validator-call-location](validator-convention.md#validator-call-location)
```

### 링크 위치

- **본문 내 참조**: 문단 끝 `> ... [file.md](file.md) 참고` 형식 인용문으로 분리
- **섹션 말미 보조 링크**: `> 상세는 [file.md](file.md) 참고` 형태로 섹션 끝에 덧붙임

## 중복 방지 원칙

### 소유권(Ownership) 기반 관리

한 규칙은 **한 문서에만 상세를 둔다**. 다른 문서는 필요 시 "참고 링크"로 포인트한다.

| 규칙 영역 | 소유 문서 |
|----------|----------|
| 레이어 전체 파일 구조 | 최상위 `{layer}-layer-guidelines.md` |
| 구성 요소별 규칙·예시·판단 기준·금지 사항·체크리스트 | 해당 `{component}-convention.md` |
| 공통 테스트·로깅 규칙 | 최상위 (하위 문서가 없으므로) |
| 의존성 방향 표 | 한쪽 선택 (예: application은 `use-case-convention`, domain은 `exception-convention`) |

### 중복이 발견되면

1. 어느 쪽이 **자연스러운 소유권**을 갖는지 결정한다 (일반적으로 구체적인 쪽).
2. 소유 문서에 내용을 유지하고, 다른 문서에서는 **한 줄 요약 + 링크**로 대체한다.
