# App 계층 — 표현 계층 전역 관심사

---

## 핵심 규칙

**표현 계층 전역 관심사 패키지는 특정 도메인 기능이 아니라 표현 계층 전체가 공유하는 횡단 관심사만 담는다.**

이 문서에서 편의상 `common/`이라는 이름을 사용하지만, 실제 프로젝트에서는 `shared/`, `global/`, `base/`, `support/` 등 다른 이름을 사용할 수 있다. 중요한 것은 이름이 아니라 역할이다.

핵심 기준은 "여러 도메인이 공유하는가"와 "표현 계층 책임인가"이다. 특정 도메인 기능에 묶이면 전역 관심사 패키지가 아니라 해당 도메인 패키지 책임이다.

이 문서는 **무엇을 표현 계층 전역 관심사로 볼 것인가**를 정의한다. 구체적인 배치 위치와 트리 구조는 [file-structure.md](file-structure.md)에서 다룬다.

---

## 전역 관심사 패키지로 분류하는 기준

전역 관심사 패키지에 둘 수 있는 대상:

- 표현 계층 설정
- 전역 예외 처리
- 공통 응답 규약
- 인증/인가 같은 표현 계층 공통 보안 처리
- HTTP 로그/MDC 같은 전역 로깅 처리
- 여러 도메인이 함께 쓰는 형식 검증기

전역 관심사 패키지에 두지 않는 대상:

- 특정 도메인 요청/응답 DTO
- 특정 도메인 Controller
- 특정 도메인 변환 로직
- 비즈니스 규칙과 도메인 정책
- DB 조회가 필요한 검증

## 대표 책임 범주

### 설정 계열

Spring MVC, Jackson, CORS, Interceptor 같은 표현 계층 설정은 `common/` 책임이다.

### 전역 예외 처리 계열

`@RestControllerAdvice`, 공통 예외 응답 변환, HTTP status 매핑처럼 여러 도메인에 공통 적용되는 예외 처리 정책은 `common/` 책임이다.

### 공통 응답 규약 계열

`BaseResponse<T>`, 공통 오류 응답, 페이지 응답 봉투처럼 응답 규약 자체는 `common/` 책임이다. 반대로 도메인별 response DTO는 각 도메인 패키지 책임이다.

### 보안 계열

Security filter chain, 인증 필터, 보안 설정처럼 표현 계층 전체에 적용되는 기술적 보안 처리는 `common/` 책임이다. 반대로 비즈니스 권한 판단은 도메인 또는 application 책임이다.

### 로깅 계열

HTTP 요청/응답 로깅, MDC 주입, 민감 정보 마스킹처럼 여러 도메인에 공통 적용되는 로깅 처리는 `common/` 책임이다.

### 공통 형식 검증 계열

전화번호, 사업자번호, 날짜 형식 같은 공통 포맷 검증기는 `common/` 책임이다. 반대로 데이터 존재 여부나 중복 여부처럼 DB 조회가 필요한 검증은 `application` 책임이다.

위 문장의 `common/`은 "이 역할을 맡는 전역 관심사 패키지"를 뜻한다. 실제 프로젝트에서 디렉토리 이름이 다르더라도 같은 책임이면 이 문서 규칙에 매핑한다.

## 판단 원칙

어떤 클래스가 전역 관심사 패키지에 들어갈지 애매하면 아래 순서로 판단한다.

1. 여러 도메인이 함께 쓰는가?
2. 표현 계층 기술 책임인가?
3. 도메인 지식 없이 독립적으로 유지 가능한가?

셋 중 하나라도 강하게 아니면 전역 관심사 패키지가 아니라 도메인 패키지 또는 다른 계층 책임일 가능성이 높다.

---

## 금지 사항

- 전역 관심사 패키지가 도메인 패키지의 클래스를 import하지 않는다.
- 도메인별 응답 타입과 비즈니스 로직을 전역 관심사 패키지에 두지 않는다.
- 비즈니스 권한 검사를 공통 보안 필터에 넣지 않는다.
- DB 조회가 필요한 검증을 공통 validator에 두지 않는다.
- 민감 정보(비밀번호, 토큰 원문)를 공통 logging 처리에서 기록하지 않는다.
- 설정 객체에 `@Service`, `@Repository` 같은 비즈니스 stereotype을 선언하지 않는다.
