# DDL Management 컨벤션

---

## 핵심 규칙

**DDL 파일은 `sql/{domain}/{table}.sql`에 두고, 엔티티 신규·변경 시 같은 커밋에서 함께 갱신한다.**

Entity와 DDL이 다른 커밋에 있으면 배포 순서에 따라 스키마 불일치가 발생한다. 이 프로젝트는 Flyway/Liquibase 없이 DDL 파일을 직접 수정하는 방식을 선택했다.

---

## 파일 위치 규칙

**규칙: DDL 파일과 seed 데이터 파일의 위치를 고정한다.**

| 파일 종류 | 위치 |
|----------|------|
| 테이블 DDL | `sql/{domain}/{table}.sql` |
| 초기 seed 데이터 | `sql/{domain}/seed.sql` |

---

## 버전 관리 방식

**규칙: 버전 접두사를 사용하지 않고, 스키마 변경 시 해당 `.sql` 파일을 직접 수정한다.**

Flyway·Liquibase 스타일의 버전 접두사(`V1__`, `V2__`)를 사용하지 않는다. 마이그레이션 히스토리 관리 없이 단순하게 스키마를 유지한다.

---

## 금지 사항

- Flyway/Liquibase 스타일의 버전 접두사(`V1__`, `V2__`)를 파일명에 사용하지 않는다.
- 엔티티 변경과 DDL 변경을 다른 커밋으로 분리하지 않는다.
- `sql/` 외부 위치에 DDL 파일을 두지 않는다.

---

## 체크리스트

- [ ] DDL 파일이 `sql/{domain}/{table}.sql` 경로에 있는가?
- [ ] 엔티티 신규 생성·변경 시 DDL을 같은 커밋에서 함께 갱신했는가?
- [ ] 파일명에 버전 접두사(`V1__` 등)를 사용하지 않았는가?
