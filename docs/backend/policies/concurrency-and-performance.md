# Concurrency And Performance Policy

## 적용 범위

- 동일 자원에 대한 동시 쓰기
- 조회 성능과 N+1 방지
- 캐시 적용과 무효화 기준
- 확장 가능성과 의도적 제약 문서화

## 핵심 원칙

- 동시 쓰기가 없으면 동시성 제어를 추가하지 않는다.
- 멀티 인스턴스 환경에서 동일 자원 경합이 발생하면 분산 락이나 낙관적 잠금 같은 명시적 제어 방식을 선택한다.
- 경합 빈도가 낮으면 낙관적 잠금과 재시도를 우선 검토하고, 경합 빈도가 높으면 분산 락을 검토한다.
- 1:N 관계 조회는 N+1을 피하도록 단일 쿼리 그룹핑 또는 단계별 조회 후 인메모리 조합을 선택한다.
- 캐시는 자주 조회되고 드물게 변경되는 리소스별 독립 데이터에만 적용한다.
- 설계 문서에는 열어둔 확장 포인트와 의도적으로 닫아둔 제약을 함께 남긴다.

## 금지 규칙

- 실시간 정합성이 필수인 데이터를 무효화 전략 없이 캐시하지 않는다.
- 페이지네이션 조회에서 무분별한 fetch join으로 중복·메모리 폭증을 만들지 않는다.
- 락 범위를 실제 경합 자원보다 넓게 잡지 않는다.
- "나중에 필요할 수 있다"는 이유만으로 과도한 추상화나 캐시를 추가하지 않는다.

## 안티패턴

- 모든 동시성 문제를 분산 락으로만 해결하려고 한다.
- 캐시 키가 충분히 세분화되지 않아 테넌트·사용자·리소스 간 데이터가 섞인다.
- 조회 성능 문제를 application 조합 로직에 숨겨 쿼리 수와 병목을 파악하기 어렵게 만든다.
- 확장 가능성만 열어 두고 해제 조건이나 현재 제약을 문서화하지 않는다.

## 코드 근거

- `application` Flow/Policy - 동시성 제어 선택 지점
- `storage` QueryDslRepository - 조회 성능과 N+1 방지
- `storage` Adapter/Repository - 캐시 무효화와 저장 경계
- `design` TDD - 확장 가능성과 의도적 제약 문서화
